# Statement Converter Backend

FastAPI backend for converting bank statement PDFs and CSV exports into a normalized CSV format.

The backend is responsible for:

- ingesting uploaded files,
- detecting whether the upload is a PDF or text-based export,
- routing known statement formats to deterministic parsers,
- falling back to a generic parser and optional LLM-assisted extraction for unknown PDFs,
- returning either a downloadable CSV or a preview payload for the frontend.

## Current scope

Supported deterministic paths:

- HDFC savings PDF
- HDFC credit card PDF
- Kotak savings PDF
- Kotak credit card PDF
- Canara savings CSV
- Generic CSV fallback

Unknown-bank handling:

- generic line-based PDF fallback
- optional Groq or Ollama LLM fallback
- structured `422` response when extraction still fails

## Tech stack

- Python 3.11+
- FastAPI
- Pydantic v2
- pydantic-settings
- dateparser
- uvicorn
- pytest
- `pdftotext` for PDF text extraction

## Setup

From the repository root:

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -e .[dev]
```

### External dependency

PDF parsing requires `pdftotext` to be installed and available on `PATH`.

Without it:

- CSV conversion still works
- PDF uploads fail during text extraction

## Run the API

```bash
cd backend
uvicorn app.main:app --reload
```

Default local URLs:

- API root: `http://127.0.0.1:8000`
- Health: `http://127.0.0.1:8000/api/health`
- Frontend shell: `http://127.0.0.1:8000/`

## Environment variables

All settings use the `STATEMENT_CONVERTER_` prefix.

| Variable | Default | Purpose |
| --- | --- | --- |
| `STATEMENT_CONVERTER_MAX_UPLOAD_BYTES` | `10485760` | Upload size limit |
| `STATEMENT_CONVERTER_MAX_PDF_PAGES` | `200` | Max PDF page count |
| `STATEMENT_CONVERTER_MAX_CSV_ROWS` | `50000` | CSV guardrail |
| `STATEMENT_CONVERTER_LLM_PROVIDER` | `none` | `none`, `groq`, or `ollama` |
| `STATEMENT_CONVERTER_GROQ_API_KEY` | unset | Required for Groq fallback |
| `STATEMENT_CONVERTER_GROQ_MODEL` | `llama-3.3-70b-versatile` | Groq model override |
| `STATEMENT_CONVERTER_OLLAMA_BASE_URL` | `http://localhost:11434` | Ollama endpoint |
| `STATEMENT_CONVERTER_OLLAMA_MODEL` | `llama3.1:8b-instruct` | Ollama model tag |
| `STATEMENT_CONVERTER_MAX_LLM_CHARS` | `20000` | Prompt truncation limit |
| `STATEMENT_CONVERTER_MIN_ROWS_THRESHOLD` | `1` | Minimum acceptable parsed rows |
| `STATEMENT_CONVERTER_LLM_REDACT_LOCAL` | `false` | Redact even before local Ollama calls |

## API surface

### `GET /api/health`

Returns a basic readiness payload:

```json
{
  "status": "ok"
}
```

### `POST /api/convert`

Accepts a multipart upload named `file` and returns normalized CSV as plain text.

Success response:

- content type: `text/csv`
- attachment filename: `normalized-statement.csv`

Failure responses:

- `422` with structured fallback payload for unsupported statements
- `422` with a simpler error payload for validation or parsing failures

### `POST /api/convert/preview`

Accepts the same multipart upload but returns JSON for the frontend preview flow.

Example shape:

```json
{
  "detected_bank": "hdfc",
  "statement_kind": "savings",
  "conversion_source": "adapter",
  "total_rows": 7,
  "preview_rows": [
    {
      "transaction_date": "06-04-2026",
      "description": "UPI PAYMENT",
      "amount": "-18838.00"
    }
  ],
  "download_token": "...",
  "download_url": "/api/download/..."
}
```

### `GET /api/download/{token}`

Streams a cached CSV generated by the preview flow.

- `404` if the token is missing or expired

## Conversion flow

### PDF flow

1. File type is detected in `app/pipeline/ingest.py`.
2. PDF text is extracted in `app/parsers/text_extractor.py`.
3. The detector in `app/parsers/detector.py` tries to identify the bank and statement kind.
4. If a deterministic parser exists, the orchestrator uses it.
5. If detection fails, no parser exists, or parsing yields too few rows, the fallback chain runs.
6. Fallback first tries a generic parser, then optionally Groq or Ollama, then returns a structured `422`.

### CSV flow

1. File is decoded as text.
2. The Canara-specific parser gets first chance.
3. If Canara does not match, the generic CSV parser is used.
4. Parsed rows are converted into the shared `Transaction` model.

## Notes on frontend integration

- If `frontend/dist` exists, FastAPI serves the built React app.
- If `frontend/dist` does not exist, FastAPI falls back to `app/static/index.html`.
- The React app depends on `POST /api/convert/preview` and `GET /api/download/{token}`.

## Known limitations

- No OCR support for scanned/image-only PDFs
- Download tokens are stored in memory only and do not survive process restarts
- LLM fallback is optional and disabled by default
- Unknown-bank generic parsing is heuristic and not a replacement for a dedicated adapter
